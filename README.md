## 이 앱이 하는 일

매주 정해진 금액(예: 50만 원)을 내 포트폴리오 종목들에 **어떻게 나눠서 살지** 자동으로 계산해 줍니다.

단순히 균등 배분하는 게 아니라, 아래 두 가지 질문에 답하면서 비중을 정합니다.

> "지금 이 종목이 시장에서 얼마나 잘 달리고 있나?" (모멘텀)  
> "지금 이 종목이 얼마나 출렁거리고 있나?" (변동성)

그리고 지금이 강세장인지 약세장인지에 따라 이 두 질문의 비중을 다르게 섞어서 최종 매수 금액을 결정합니다.

## 전략 철학 — 왜 이 방식인가?

### 💡 적립식 투자의 함정

매월 정액을 투자하는 적립식(DCA, Dollar Cost Averaging)은 훌륭한 습관이지만, "어떤 종목을 얼마나 살지"를 무작위로 결정하면 수익률 차이가 크게 납니다. 이 앱은 적립식의 규칙성은 유지하면서, 매수 배분은 데이터 기반으로 최적화합니다.

### 💡 왜 모멘텀인가?

수십 년간의 글로벌 실증 연구(Jegadeesh & Titman, 1993; Asness et al., 2013)에서 반복적으로 확인된 사실이 있습니다.

> 최근 6 ~ 12개월간 잘 오른 종목은, 향후 3 ~ 6개월도 계속 오르는 경향이 있다.

이를 **모멘텀 팩터(Momentum Factor)** 라고 합니다. 단순한 추세 추종이 아니라, 기업의 펀더멘털 개선이 주가에 반영되는 속도 차이(정보 확산 지연)를 이용하는 전략입니다.

### 💡 왜 저변동성인가?

반직관적이지만, 주가가 덜 출렁이는 종목이 장기적으로 더 좋은 위험조정수익률(Sharpe Ratio)을 냅니다. 이를 **저변동성 이상현상(Low Volatility Anomaly)** 이라 합니다. 특히 하락장에서 낙폭이 작아 원금 보전에 유리합니다.

### 💡 왜 두 팩터를 섞는가? (Tilt 방식)

모멘텀은 강세장에서 강하지만 급락장에선 크게 손실이 납니다(모멘텀 크래시). 저변동성은 하락장 방어에 탁월하지만 강세장에선 시장에 뒤처집니다. 두 팩터는 서로의 약점을 보완합니다. 시장 국면에 따라 비율을 동적으로 조절하면 단독 팩터보다 더 안정적인 수익 곡선을 만들 수 있습니다.

## 알고리즘 상세 해설

### Step 1: 시장 국면 판단 (강세장 vs 약세장)

**기준 지표: QQQ(나스닥 100 ETF)의 200일 이동평균선**

```
현재 QQQ 가격 > QQQ 200일 이동평균  →  강세장 (Bull Market)
현재 QQQ 가격 < QQQ 200일 이동평균  →  약세장 (Bear Market)
```

**왜 QQQ인가?**  
글로벌 주식시장, 특히 미국 기술주와 성장주에 투자하는 개인 투자자에게 QQQ는 가장 대표적인 위험선호 지수입니다. QQQ가 장기 평균 위에 있다는 것은 "시장 참여자들이 전반적으로 위험을 감수하고 있다"는 신호입니다.

**왜 200일 이동평균인가?**  
200일(약 10개월)은 단기 노이즈를 걸러내면서도 추세 전환을 빠르게 포착하는 황금 구간입니다. 글로벌 헤지펀드와 기관 투자자들이 공통으로 사용하는 기준이기 때문에, 이 선이 지지/저항으로 작동하는 자기실현적 효과도 있습니다.

---

### Step 2: 모멘텀 점수 산출

단일 기간이 아닌 **4개 기간을 동시에 측정**한 뒤 가중 평균합니다.

| 기간 | 영업일 기준 | 의미 | 가중치 |
|------|------------|------|--------|
| 단기 | 21일 (약 1개월) | 최근 추세 | 낮음 |
| 중단기 | 63일 (약 3개월) | 분기 추세 | 중간 |
| 중기 | 126일 (약 6개월) | 반기 추세 | 높음 |
| 장기 | 252일 (약 12개월) | 연간 추세 | 높음 |

**계산 방식 — 수익률 기반 순위화 (Rank Normalization)**

1. 각 기간에서 포트폴리오 내 모든 종목의 수익률을 계산
2. 수익률이 높을수록 높은 순위(점수)를 부여
3. 순위를 0~1 사이로 정규화 (최하위 = 0, 최상위 = 1)
4. 4개 기간의 정규화된 순위를 가중 평균

**왜 수익률이 아닌 순위를 쓰는가?**  
수익률 자체는 종목마다 스케일이 다릅니다(예: +5% vs +50%). 순위를 쓰면 종목 간 비교가 공정해지고, 극단적 이상값(outlier)이 전체 비중을 왜곡하는 것을 방지할 수 있습니다. 이는 크로스 섹션 모멘텀 전략의 표준 처리 방식입니다.

---

### Step 3: 저변동성 점수 산출

**기준: 60일(약 3개월) 일간 수익률의 표준편차**

```
변동성 점수 = 1 / (각 종목의 표준편차)
```

표준편차가 낮을수록(덜 출렁일수록) 점수가 높아집니다. 이 점수도 모멘텀과 마찬가지로 순위 기반으로 0~1 사이에 정규화됩니다.

**왜 60일 변동성인가?**  
너무 긴 기간(1년)의 변동성은 현재 시장 상황을 반영하지 못합니다. 너무 짧은 기간(1주)은 노이즈가 많습니다. 60일(약 3개월)은 현재의 변동성 체제를 안정적으로 반영하는 구간입니다.

### Step 4: 팩터 합산 & 비중 결정

**시장 국면에 따른 팩터 가중치 (Tilt)**

| 시장 국면 | 모멘텀 비중 | 저변동성 비중 | 전략적 의미 |
|----------|------------|--------------|-----------|
| 🐂 강세장 | **70%** | 30% | 상승 추세 종목에 집중 투자 |
| 🐻 약세장 | 40% | **60%** | 방어적 종목으로 포트폴리오 보호 |

**합산 점수 계산**

```
강세장: 최종 점수 = 모멘텀 점수 × 0.7 + 저변동성 점수 × 0.3
약세장: 최종 점수 = 모멘텀 점수 × 0.4 + 저변동성 점수 × 0.6
```

**시가총액 가중 (선택)**

옵션을 켜면 최종 점수에 각 종목의 시가총액을 추가로 반영합니다.

```
조정 점수 = 최종 점수 × 시가총액 비율
```

- **매수 추천 탭**: 현재 시총을 그대로 사용합니다 (현재 시점 의사결정이므로 적절).
- **백테스트 탭**: `발행주수(현재) × 해당 시점 주가`로 과거 시총을 근사합니다. 순수 현재 시총을 과거에 그대로 적용하는 것보다 look-ahead bias를 크게 줄일 수 있습니다.

**최종 비중 결정 규칙**

1. 조정 점수를 기반으로 비중을 계산합니다: `비중 = 각 종목 점수 / 전체 점수 합계`
2. **종목당 최대 비중 25% 캡을 적용합니다.** 25%를 초과하는 비중은 잘라내고, 남은 금액을 나머지 종목에 재배분합니다.
3. 이 과정을 모든 종목이 25% 이하가 될 때까지 반복합니다.

**왜 25% 캡인가?**  
단일 종목 집중 리스크를 막기 위한 분산투자 원칙입니다. 아무리 좋은 모멘텀 점수를 받은 종목이라도 전체 포트폴리오의 1/4 이상을 차지하면 해당 종목 단독 이슈로 전체 포트폴리오가 크게 흔들릴 수 있습니다.

---

### Step 5: 매수 금액 계산

**팩터 비중 단순 비례 배분 (Factor-Proportional Allocation)**

팩터 알고리즘이 산출한 목표 비중을 그대로 주간 예산에 곱해 매수 금액을 결정합니다.

```
매수 금액 = 목표 비중 × 주간 예산
```

**예시 (주간 투자금 50만 원, 3개 종목)**

| 종목 | 팩터 목표 비중 | 이번 주 매수금액 |
|------|--------------|----------------|
| AAPL | 50% | ₩25만 |
| MSFT | 30% | ₩15만 |
| NVDA | 20% | ₩10만 |

매주 팩터 신호를 그대로 예산 배분에 반영하므로, 알고리즘의 판단이 왜곡 없이 일관성 있게 적용됩니다.

**왜 단순 비례 배분인가?**  
매도 없는 적립식 구조에서 "부족분 채우기" 방식은 잔고가 커질수록 주간 예산의 영향력이 희석되어 팩터 신호가 점점 무력화됩니다. 단순 비례 배분은 매주 팩터 신호를 충실하게 반영하고, 벤치마크와 동일하게 매주 예산 전액을 투자하므로 공정한 비교가 가능합니다.

## 기기별 데이터 개별화 & 영구 저장

### 작동 방식

접속 시 브라우저 URL에 고유 UUID가 자동 부여됩니다.

```
https://yourapp.streamlit.app?uid=abc-123-...
```

이 UUID를 키로 **Supabase(클라우드 DB)** 에 포트폴리오를 저장하므로, 앱이 재시작되거나 sleep 상태에서 깨어나도 데이터가 영구 보존됩니다.

```
접속  →  uid 생성  →  Supabase에서 내 포트폴리오 로드
저장  →  Supabase에 즉시 반영  →  앱 재시작해도 유지 ✅
```

### 환경별 저장 방식

| 환경 | 저장 방식 | 영구 보존 |
|------|----------|----------|
| **Streamlit Cloud** | Supabase DB | ✅ |
| **로컬 실행 + secrets.toml** | Supabase DB | ✅ |
| **로컬 실행 (secrets 없음)** | `data/portfolio_{uid}.json` | ✅ (로컬 파일) |

### 기기 간 데이터 공유

같은 URL(`?uid=xxxx` 포함)을 다른 기기에서 열면 동일한 포트폴리오에 접근할 수 있습니다.

```
📱 모바일  ──┐
💻 PC      ──┼──  ?uid=abc-123  ──→  Supabase  →  동일 데이터
🖥️ 태블릿  ──┘
```

> ⚠️ **URL 북마크 필수**: 새 탭으로 접속하면 새 UUID가 생성됩니다. 처음 접속 후 URL을 반드시 북마크하거나 홈 화면에 추가하세요.

### 기본 포트폴리오 (MAG7)

처음 접속하는 신규 uid는 Magnificent 7 종목이 0주로 자동 세팅됩니다.

| 티커 | 종목명 |
|------|--------|
| AAPL | Apple |
| MSFT | Microsoft |
| NVDA | NVIDIA |
| AMZN | Amazon |
| GOOGL | Alphabet |
| META | Meta |
| TSLA | Tesla |

종목을 추가하거나 삭제해 본인만의 포트폴리오로 자유롭게 수정할 수 있습니다.

### Supabase 설정

```toml
# .streamlit/secrets.toml  ← 절대 git에 올리지 마세요!
SUPABASE_URL = "https://your-project.supabase.co"
SUPABASE_KEY = "your-publishable-key"
```

Streamlit Cloud 배포 시에는 대시보드 **Settings → Secrets** 에 동일한 내용을 입력합니다.

## 백테스트 해석 주의사항

백테스트 결과는 전략 이해를 돕기 위한 참고용입니다. 아래 세 가지 한계를 반드시 인지하고 해석하세요.

### ⚠️ 1. 생존 편향 (Survivorship Bias)

백테스트는 **현재 보유 중인 종목(살아남은 종목)** 만으로 과거를 시뮬레이션합니다. 과거에 상장폐지되거나 크게 하락한 종목은 포함되지 않으므로, 실제 성과보다 과대평가될 수 있습니다.

### ⚠️ 2. 시가총액 근사 오차

yfinance는 과거 시가총액 데이터를 제공하지 않습니다. 백테스트에서는 `현재 발행주수 × 과거 주가`로 시총을 근사합니다. 발행주수는 수년간 크게 변하지 않으므로 순수 현재 시총을 쓰는 것보다 look-ahead bias가 훨씬 적지만, 자사주 매입·유상증자 등으로 인한 오차는 일부 잔존합니다.

### ⚠️ 3. 수익률 지표: CAGR이 아닌 XIRR 사용

적립식 투자에서 CAGR은 부적합한 지표입니다. 매주 나눠 투자하는 구조에서 CAGR은 "모든 돈을 첫날 한 번에 넣었다"고 가정해 수익률을 과대평가합니다.

이 앱은 **XIRR(내부수익률)** 을 사용합니다.

| 지표 | 계산 방식 | 적립식 적합성 |
|------|----------|-------------|
| CAGR | `(최종금액/총투자금)^(1/기간) - 1` | ❌ 과대평가 |
| XIRR | 각 투자 시점과 금액을 반영한 역산 연수익률 | ✅ 정확 |

> **예시**: 3년 적립, 총 1,560만 원 투입 → 2,000만 원 평가  
> CAGR: **+28.2%** (과대) vs XIRR: **+16.8%** (실제)

## 설치 및 실행

```bash
# 1. 저장소 클론
git clone https://github.com/mavro7910/quant_portfolio_web.git
cd quant_portfolio_web

# 2. 의존성 설치
pip install -r requirements.txt

# 3. (선택) Supabase 연동 시 secrets 설정
mkdir .streamlit
cat > .streamlit/secrets.toml << EOF
SUPABASE_URL = "https://your-project.supabase.co"
SUPABASE_KEY = "your-publishable-key"
EOF

# 4. 실행
streamlit run app.py
```

브라우저에서 `http://localhost:8501` 로 자동 열립니다.  
같은 Wi-Fi의 스마트폰에서는 `http://<PC_IP>:8501` 로 접속하세요.

## 클라우드 배포

### ✅ Streamlit Community Cloud (권장, 무료)

1. 이 레포를 GitHub에 push  
   (`.streamlit/secrets.toml`은 `.gitignore`에 추가해 올리지 마세요)
2. [share.streamlit.io](https://share.streamlit.io) 접속 후 로그인
3. **New app** → 레포/브랜치/`app.py` 선택 → **Deploy**
4. 앱 **Settings → Secrets** 에 아래 내용 입력:
   ```toml
   SUPABASE_URL = "https://your-project.supabase.co"
   SUPABASE_KEY = "your-publishable-key"
   ```
5. 생성된 URL 접속 후 `?uid=xxxx` 포함된 주소를 북마크

> `requirements.txt`는 반드시 **프로젝트 루트**에 있어야 자동 설치가 됩니다.  
> 일정 시간 접속이 없으면 sleep 상태로 전환되며, 첫 접속 시 30~60초 재시작 시간이 걸릴 수 있습니다.  
> Supabase 연동 덕분에 sleep 후 깨어나도 포트폴리오 데이터는 유지됩니다.
